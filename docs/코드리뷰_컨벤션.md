## 📌 1. 코드 리뷰의 목적

코드 리뷰는 단순 검수가 아니라 아래와 같은 **목적과 가치**를 달성하기 위한 활동으로 정의합니다.

✔ 코드 품질 및 일관성 향상

✔ 잠재적 버그·설계 오류 사전 발견

✔ 지식 공유 및 팀 전체 역량 향상

✔ 개발 문화 강화 및 커뮤니케이션 품질 향상

✔ 코드베이스 건강성(code health) 유지

## 📌 2. 프로세스 규칙

### 2-1. PR 단위는 작게

- PR/CL은 **작은 단위(일관된 한 기능 또는 변경)**로 작성합니다.
- Reviewer가 빠르게 이해 가능하고 병목이 되지 않는 범위로 유지합니다.
- 100~300줄 내외가 이상적인 가이드라인. (Google은 100~1000 라인 권장 범위 언급)

### 2-2. 비동기 커뮤니케이션 우선

- GitHub PR과 같은 비동기 도구를 기본으로 하여 리뷰 진행
- Slack/이슈 등 추가 컨텍스트를 PR에 명시적으로 첨부합니다.
- 충분한 문맥 없이 리뷰하지 않습니다.

### 2-3. 우선순위 라벨링

- PR에는 **우선순위(D-n)** 라벨을 붙여 리뷰 우선순위를 명시합니다.
- D-0: 긴급 (장애 / 빌드 실패)
- D-2: 2일 이내 검토 요청
- 기타: 상황에 따라 D-N 으로 처리 (사회적 컨텍스트 제공)

## 📌 3. 코드 리뷰 체크리스트

다음은 리뷰 시 반드시 확인하는 **공통 체크리스트**입니다 (Google Eng-Practices 반영).

### 3-1. 설계 (Design)

- 기존 설계와의 일관성
- 확장성과 유지보수성 고려 여부
- 복잡도 최소화 여부 (simplification)

### 3-2. 기능 (Functionality)

- 요구사항에 적합한 로직인지
- 오류/예외케이스 처리가 충분한지
- 기능이 실제 사용자 사용 시 의도대로 동작하는지

### 3-3. 테스트

- 관련 테스트가 포함되어 있는가
- 테스트 커버리지 및 경계조건이 충분한가

### 3-4. 네이밍 (Naming)

- 명확하고 일관된 이름인가
- 도메인/비즈니스 의미를 정확히 반영하는가

### 3-5. 문서화/주석

- 중요한 로직은 필요한 설명이 있는가
- 너무 과하거나 불필요한 주석은 없는가
- PR 설명에 변경 이유/맥락이 구조적으로 충분한가 (Google CL 설명 규칙 참조)

### 3-6. 스타일 (Style)

- 조직/언어별 컨벤션을 준수하는가
- 코딩 포맷팅은 자동화된 도구로 해결하며 리뷰 에서 제외하는 것이 효율적임 (뱅크샐러드 사례)

## 📌 4. 리뷰어의 자세 (Communication & Tone)

성공적인 코드 리뷰는 기술 검토뿐 아니라 커뮤니케이션 방식도 중요합니다.

### 📌 기본 원칙

✔ 이유(why)를 설명

✔ 제안(respectful suggestion), 명령/지시가 아닌 **대안 제시**

✔ 적극적인 경청 및 쟁점 합의 유도

✔ 리뷰의 동반성장 목적 강조

✔ 필요 시 칭찬 및 긍정 피드백 제공 (LGTM 또는 Encouragement)

### 🧠 리뷰 코멘트 포맷

- 잘못된 지점 → 설명 → 대안/추천
- 객관적 팩트 기반 → 주관적 의견 최소화
- 분쟁 시 합의 지향 (Consensus first)
